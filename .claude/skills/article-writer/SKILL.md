# 公众号技术文章写作

将技术素材改写为高质量原创公众号文章，面向开发者群体，风格为**专业实战，适度幽默**。

## 爆款底层逻辑：赛马机制

公众号采用"社交+算法"双重传播机制。文章发布后先推给少量用户（几百人），系统根据**完读率、点赞率、转发率、评论数**四大核心指标决定是否推入更大流量池（赛马机制）。胜出的文章不断进入下一个流量池，最终实现10万+、100万+阅读量。

### 赛马机制的四大核心指标

| 指标 | 写作策略 | 底层逻辑 |
|:-----|:---------|:---------|
| 完读率 | 开幕雷击抓住前 3 秒 → 短段落降低阅读压力 → 中间埋锚点防跳出 → 控制总字数在 1500-2500 字 | 完读率高说明内容质量好，系统会推给更多人 |
| 点赞率 | 金句收尾引发共鸣 → 观点鲜明让人想"表态" → 提供实用价值让人想"收藏" | 点赞数高说明用户喜欢，是强推荐信号 |
| 转发率 | 内容有社交货币价值（转发后显得有见识/有态度） → 标题无转发心理负担 → 包含可截图转发的表格/金句/Checklist | 转发是最强的推荐信号，会带来指数级传播 |
| 评论数 | 结尾抛出争议性问题 → 引导读者站队讨论 → 设置开放性技术话题 | 评论区讨论热烈说明文章有话题性 |

### 如何利用赛马机制

**1. 寻找蓝海领域**：少跟随主流技术热点，积极寻找细分领域（如特定框架的冷门特性、小众但实用的工具）。即使小众也无所谓，竞争少更容易胜出，14亿人口中再小众的技术领域都有庞大的目标群体。

**2. 字数控制的赛马逻辑**：960字起步可插两条广告，但不要写太长。1500-2500字是最佳区间，既能插入足够广告，又能保持高完读率。超过3000字完读率会断崖下跌，在赛马中直接出局。

**3. 与对标差异化竞争**：
- 对标笼统写，你就详细写（如对标只给配置，你给配置+原理+踩坑经验）
- 对标三个角度，你就五个角度且更生动
- 对标标题平庸但爆了，你就用更强的标题公式
- 人无我有，人有我优，在赛马中才能胜出

**4. 发布时机优化**：和对标的发布时间稍微错开，减少直接竞争。如对标晚上8点发，你就晚上7点或9点发。可在微信公众号后台"内容分析"模块查看自己账号的最佳发布时间。

## 核心写作准则

以资深开发者的第一人称视角写作，像跟同事复盘线上事故一样讲技术。遵守以下五条铁律：

1. **开幕雷击**：文章前 100 字必须有"雷击点"——一个让读者停不下来的冲突、悬念或反差。技术文章可用线上事故、性能数据反差、反直觉结论作为雷击点。**禁止开门见山式开头**（除非干货极其突出），要制造悬念让读者忍不住往下看。不要杜撰虚假事件，除非是事故类引出的思考
2. **数据说话**：凡提到"优化""提升""高效"，必须给出性能对比数据（QPS、延迟、内存等），没有真实数据时用合理估算并标注
3. **一文一核心**：每篇文章聚焦一个核心知识点，不要贪多。读者记住一个点比模糊记住十个点更有传播力
4. **去 AI 味**：禁用"综上所述""值得注意的是""此外""不仅如此""在当今时代"等 AI 废话，详见 [references/style-guide.md](references/style-guide.md)
5. **资产 100% 保留**：原文中的图片、架构图、流程图必须原封不动保留在对应位置

## 执行流程

按以下四个阶段顺序执行，每完成一个阶段向用户汇报进度。

### 阶段一：收集素材与需求

向用户确认以下信息（缺什么问什么，不要一次问太多）：

| 必要信息 | 说明 |
|:---------|:-----|
| 素材来源 | URL 链接、本地文件路径、或直接粘贴的文本 |
| 技术主题 | 文章核心技术点（如 Redis、Kafka、Go 并发等） |
| 目标读者 | 初级开发 / 中级开发 / 高级开发 / 架构师 |

信息齐全后，获取并通读素材内容，提取所有图片资产的 URL 或路径。同时分析素材中最具"爆点"的技术观点——反直觉的结论、惊人的性能数据、常见的认知误区，这些将成为标题和开头的核心素材。

### 阶段二：生成标题

参考 [references/title-templates.md](references/title-templates.md) 中的爆款标题公式，生成 **5 个候选标题**，要求：

- 覆盖至少 3 种不同类型（冲突反转、极限场景、经验自嘲、利益直击、大厂背书）
- 标题控制在 25 字以内，关键信息前置
- 考虑读者转发朋友圈的场景：标题不能让转发者有心理负担，转发后要显得"有见识"
- 至少 1 个标题包含具体数字（数字比文字更抓眼球）
- 展示给用户选择，确认后进入下一阶段

### 阶段三：撰写正文

根据素材内容和目标读者，按"爆款结构"组织文章。参考 [references/structure-guide.md](references/structure-guide.md) 了解结构模板，参考 [references/style-guide.md](references/style-guide.md) 了解风格要求。

**爆款结构（7 个模块，按需裁剪）：**

| 模块 | 核心任务 | 字数建议 | 是否必选 |
|:-----|:---------|:---------|:---------|
| 开幕雷击 | 前 100 字用冲突/数据/悬念抓住读者，让人无法划走 | 50-100 字 | 必选 |
| 痛点共鸣 | 用具体生产场景描述读者的真实痛点，制造"我也遇到过"的代入感 | 100-200 字 | 必选 |
| 概念破冰 | 用接地气的类比定义核心概念，降低认知门槛 | 100-200 字 | 必选 |
| 风险预警 | 指出"不这么做"的严重后果，制造紧迫感 | 100-200 字 | 必选 |
| 技术实现 | 提供生产级代码实现，包含错误处理和关键注释 | 300-500 字 | 必选 |
| 性能压测 | 提供优化前后的数据对比表格 | 100-200 字 | 仅性能类 |
| 钩子收尾 | 避坑指南 + 金句 + 互动问题，驱动点赞/转发/评论 | 150-300 字 | 必选 |

**总字数控制在 1500-2500 字**。技术文章不宜太短（缺乏深度）也不宜太长（影响完读率和赛马竞争）。

**写作要点：**

- 每个小标题要有创意和吸引力，避免"一、二、三"式的干巴标题
- 段落短小精悍，**手机屏幕不超过 3-6 行，遵循"一句话一段，最多三句话一段"原则**。短段落是完读率的生命线
- 每 300-500 字埋一个"锚点"（悬念、反转、金句），防止读者中途跳出。锚点是勾住读者继续往下看的关键词或句子，如"接下来我要说的这个方法，可能会颠覆很多人的认知..."
- 代码必须包含生产级注释，说明"为什么这么写"和"不这么写会怎样"
- 复杂配置或选型对比使用 Markdown 表格，表格比文字更容易被读者截图转发（提升转发率）
- 根据目标读者调整深度：初级读者多解释原理，高级读者多讲架构决策
- 文章中间可以穿插 1-2 句轻松的吐槽或类比，但不要刻意搞笑，保持专业底色
- 在结尾引导互动：抛出 1-2 个开放性或争议性技术问题，引导读者在评论区讨论（提升评论数）

### 阶段四：自检与交付

对照 [references/review-checklist.md](references/review-checklist.md) 逐项自检，修正不合格项后输出完整 Markdown 文章。

**交付物：**

- 完整 Markdown 文件，包含确认后的标题和正文
- 所有原文图片资产 100% 保留
- 保存到当前工作目录
